# Инструкция для ESP32: подключение к серверу умного дома в локальной сети

Ты и владелец сервера должны быть в **одной Wi‑Fi сети** (один роутер / одна подсеть).

---

## 1. Узнать IP компьютера с сервером

Пусть друг скажет тебе **локальный IP** своего ноутбука/ПК в Wi‑Fi. Примеры:

- `192.168.1.105`
- `192.168.0.42`
- `10.0.0.15`

**Как он может посмотреть (Linux):** `ip a` или `hostname -I`  
**Windows:** `ipconfig` → IPv4-адрес Wi‑Fi адаптера.

Дальше в примерах вместо `IP_СЕРВЕРА` подставляй этот адрес.

---

## 2. Адрес API

Сервер слушает порт **8000**.

**Базовый URL:**

```text
http://IP_СЕРВЕРА:8000
```

**Два эндпоинта с одним путём `/api/update`:**

| Метод | Назначение |
|-------|------------|
| **POST** | Отправить показания датчиков (тело JSON). В ответе только `received`. |
| **GET** | Запросить текущее состояние устройств и сценарий (`devices`, `scenario`, `auto_mode`). Тела нет. |

```text
POST http://IP_СЕРВЕРА:8000/api/update
GET  http://IP_СЕРВЕРА:8000/api/update
```

Для POST заголовок: `Content-Type: application/json`

Проверка с телефона/ноутбука в той же сети: открыть в браузере  
`http://IP_СЕРВЕРА:8000/docs` — должна открыться документация Swagger.

Если у друга фронт за nginx (docker compose), сайт может быть на **порту 8080**, Swagger тогда: `http://IP_СЕРВЕРА:8080/docs`. Для **ESP** удобно держать API на **порту 8000** (`POST`/`GET` `...:8000/api/update`).

---

## 3. Что отправлять (один JSON за запрос)

Тело запроса — **объект** с одним полем:

| Поле | Что это |
|------|---------|
| `readings` | **Массив** показаний. Каждый элемент — один датчик в одной комнате. |

Время для всех записей этого запроса **проставляет сервер** (момент приёма, UTC) и сохраняет в БД — передавать `timestamp` с ESP **не нужно**.

Каждый элемент в `readings`:

| Поле | Тип | Описание |
|------|-----|----------|
| `sensor_name` | строка | Имя датчика **строго из списка ниже** (латиница, как в таблице). |
| `room_id` | целое число | Номер комнаты: **1** или **2** (у вас две одинаковые комнаты). |
| `value` | число | Для «числовых» датчиков — обычное значение. Для «да/нет» — **только `0.0` (выкл/нет) или `1.0` (вкл/да)**. |

### Разрешённые имена датчиков (`sensor_name`)

**Числа (float):**

- `temperature` — температура (например `22.5`)
- `humidity` — влажность (например `45.0`)
- `light_level` — освещённость в **люксах** (обычно только комната **1**; сервер использует это в логике сценариев)

**Да/нет (bool, на сервере как число):**

- `movement` — движение
- `gas` — газ / датчик газа

Состояние ламп (`night_light`, `day_light` как **устройства**) задаётся **с сервера** — смотри **`GET /api/update`** → поле `devices`; в `readings` их **не** передают.

**Не отправляй в `readings`:** `vent`, `socket_tv`, `socket_ac`, `servo`, `rgb` — это **устройства**, их состояние приходит **только в ответе** в массиве `devices` (управление с сайта).  
**`servo`** — штора, **только комната 2** (`device_type: "servo", room_id: 2`).  
**`rgb`** — RGB-подсветка, **только комната 1** (`device_type: "rgb", room_id: 1`); в API пока только вкл/выкл (`is_on`).

Других имён в `sensor_name` быть не должно — сервер вернёт ошибку **400**.

### Пример полного тела запроса

```json
{
  "readings": [
    {"sensor_name": "temperature", "room_id": 1, "value": 23.1},
    {"sensor_name": "humidity", "room_id": 1, "value": 48.0},
    {"sensor_name": "light_level", "room_id": 1, "value": 320.0},
    {"sensor_name": "movement", "room_id": 1, "value": 0.0},
    {"sensor_name": "gas", "room_id": 1, "value": 0.0},
    {"sensor_name": "temperature", "room_id": 2, "value": 21.8},
    {"sensor_name": "humidity", "room_id": 2, "value": 52.0}
  ]
}
```

Можно слать **не все** датчики за раз — только те, что реально подключены; главное — корректные `sensor_name` и `room_id`.

## 4. Ответы сервера

### После POST (только подтверждение приёма)

```json
{ "received": 7 }
```

- **`received`** — сколько записей из `readings` сохранено.

Полный список устройств после POST **не** возвращается — его нужно запросить отдельно (см. ниже).

### GET — актуальное состояние (опрос чаще, чем POST)

Запрос: **`GET /api/update`** без тела. Успешный ответ (пример):

```json
{
  "scenario": "sleep",
  "auto_mode": true,
  "devices": [
    {"device_type": "night_light", "room_id": 1, "is_on": false},
    {"device_type": "night_light", "room_id": 2, "is_on": false},
    {"device_type": "day_light", "room_id": 1, "is_on": false},
    {"device_type": "day_light", "room_id": 2, "is_on": false},
    {"device_type": "vent", "room_id": 1, "is_on": false},
    {"device_type": "vent", "room_id": 2, "is_on": false},
    {"device_type": "socket_tv", "room_id": 1, "is_on": true},
    {"device_type": "socket_tv", "room_id": 2, "is_on": false},
    {"device_type": "socket_ac", "room_id": 1, "is_on": false},
    {"device_type": "socket_ac", "room_id": 2, "is_on": false},
    {"device_type": "rgb", "room_id": 1, "is_on": false},
    {"device_type": "servo", "room_id": 2, "is_on": false}
  ]
}
```

- **`scenario`** — текущий сценарий дома (`sunny`, `cloudy`, `night`, `sleep`).
- **`auto_mode`** — если `true`, при последнем приёме датчиков по POST сервер мог пересчитать устройства по сценарию; если `false`, состояние задаётся вручную с сайта.
- **`devices`** — **желаемое состояние** выходов. ESP выставляет реле/розетки по `is_on` для пары **`device_type` + `room_id`**:
  - `night_light`, `day_light` — свет
  - `vent` — вентиляция
  - `socket_tv` — розетка под телевизор
  - `socket_ac` — розетка под кондиционер
  - `servo` — штора, **только `room_id: 2`**
  - `rgb` — RGB-лента, **только `room_id: 1`**; `is_on` — вкл/выкл ленты (цвет задаётся в прошивке отдельно, если нужно)

Имеет смысл **реже** слать **POST** с показаниями (например раз в 10–60 с), а **GET /api/update** вызывать **чаще** (например раз в 1–3 с), чтобы быстро подхватывать смену `is_on` без лишней передачи датчиков.

## 5. Частые проблемы

| Проблема | Что проверить |
|----------|----------------|
| Не коннектится | Один Wi‑Fi, верный `IP_СЕРВЕРА`, порт `8000`, firewall на ПК разрешает вход на 8000. |
| 400 Bad Request | Имя датчика не из списка, опечатка в `sensor_name`, `room_id` не число или &lt; 1. |
| Connection refused | Docker не запущен или порт не проброшен на хост. |

---

## 6. Краткая шпаргалка

1. Wi‑Fi тот же, что у друга с сервером.  
2. Показания: **POST** `http://IP_СЕРВЕРА:8000/api/update`, JSON.  
3. Состояние реле: **GET** тот же URL `/api/update`.  
4. В `readings`: только поля `sensor_name`, `room_id`, `value`.  
5. Имена датчиков — **только** из раздела выше.  
6. Булевы датчики: **0.0 / 1.0**.  
7. Комнаты: **1** и **2**.  
8. После **GET** смотреть **`devices`** и выставлять реле под `is_on`.

Если что-то не сходится — пусть друг откроет `http://IP_СЕРВЕРА:8000/docs`: там **GET** и **POST** `/api/update`.
